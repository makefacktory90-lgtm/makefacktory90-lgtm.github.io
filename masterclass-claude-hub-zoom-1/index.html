SEGMENT 01 // CORE PROTOCOL

6 шагов pipeline

Если задача сложнее «напиши пост» — кнопки нет. Есть маршрут. Шесть шагов, отработанная последовательность. Шорткаты дают игрушки, последовательность даёт инструмент.

Промпты по каждому шагу — отработанные у Сергея Риса, курс vibecoding.phd. Один из лучших фидеров по вайб-кодингу в России.

01

Идея // надиктовка голосом

Открываем Wispr Flow или голос в ChatGPT/Claude и просто наговариваем идею. Не пишем — говорим. Просим агента быть собеседником, а не исполнителем: «помоги докрутить мою идею через интервью».

PROMPT.01 // БАЗОВЫЙ
У меня в голове идея: [своими словами]. Помоги её кристаллизовать через интервью. Задай мне 5–7 вопросов с тремя вариантами ответов, по одному. Вопросы задавай строго по одному и дожидайся моего ответа. В конце дай мне структурированный промпт в формате markdown.
02

Промпт // версия с экспертами

Когда хочется не вариться в собственной голове — просим агента подключить «комментарии экспертов» к каждому вопросу. Внешний контекст, чужие точки зрения, рамки. Часто именно тут идея ломается и собирается заново.

PROMPT.02 // С ЭКСПЕРТАМИ
У меня в голове идея: [своими словами]. Помоги её кристаллизовать через интервью. Задай мне 5–7 вопросов с тремя вариантами ответов, по одному. С каждым вопросом давай комментарии экспертов, которые могли бы это обсудить, и что бы они сказали — внешний контекст, чтобы я не варился в своей голове. Вопросы задавай строго по одному и дожидайся моего ответа. В конце дай мне структурированный промпт в формате markdown.
03

PRD // продакт-менеджер челленджит

Идём в формат «разговор с product manager». Нейронка вытягивает наружу: цель, ценность, юзеры, сценарии, KPI, риски. На выходе — реальное продуктовое ТЗ. Формат — markdown (нейронки лучше понимают). Максимум 2 страницы — иначе теряется фокус.

PROMPT.03 // PRD
Вот идея: [описание]. Создай PRD со следующей структурой: Purpose, Target Users (персоны), User Scenarios (3–5), Success Metrics, Constraints, Risks. Формат: markdown, максимум 2 страницы.
04

Архитектура // карта проекта

Из PRD строим минимальную карту: что где лежит, какие сущности в базе, какие endpoints наружу. Если задача серьёзная — одного Claude не хватит, заранее закладываем внешние API. Чем чётче карта — тем меньше блужданий на шаге кода.

PROMPT.04 // ARCHITECTURE
Вот PRD: [вставить]. Предложи минимальную архитектуру. Нужно: 1) Структура папок, 2) Описание каждого файла (1 строка), 3) Data model, 4) Список API endpoints. Стек: [constraint из PRD].
05

Код // агент кодит сам

Отдаём всё предыдущее в Claude Code или Codex. Не отдаём всё сразу — делим на чанки и подаём по одному, с релевантным контекстом. На каждом куске уточняем у модели «что нужно для следующего шага» — она сама ведёт.

Если что-то сломалось — отдельный промпт для диагностики (ниже).

PROMPT.05 // CODE
Роль: Fullstack developer. Задача: [конкретный чанк функционала]. Контекст: [только релевантные файлы]. Когда закончишь — скажи что нужно для следующего.
PROMPT.05B // DEBUG
Ошибка: [полный текст ошибки]. Как воспроизвести ошибку — среда, тулы и т.д. Код: [релевантный фрагмент]. Что я уже пробовал: [список]. Объясни вероятную причину и предложи 2–3 варианта решения с объяснением, почему каждый может помочь.
06

Деплой // онлайн или локально

Если сервис только для внутреннего использования — можно жить локально, запускать командой прямо в Claude или Codex. Если онлайн — Vercel (бесплатно) или GitHub Pages. Регистрация руками один раз, дальше агент сам пушит.

Сразу просим пошаговую инструкцию — с env vars, билдом и планом отката. Если что-то сломается ночью — план отката уже готов.

PROMPT.06 // DEPLOY
Проект: [стек]. Нужно задеплоить на [платформа]. Создай пошаговую инструкцию. Укажи: 1) Какие env vars нужны, 2) Команды для билда, 3) Как проверить что работает, 4) Как откатить если сломается.

SEGMENT 03 // KEY PRINCIPLES

Три отличия от обычной нейронки

PRINCIPLE.01

Кнопок нет

Готовых правильных решений не существует. Вайб-кодингу чуть больше года. Есть только маршруты других — их берут и адаптируют под себя. Один-в-один не получится никогда.

PRINCIPLE.02

Советоваться, а не приказывать

Не «сходи и возьми это». А «у меня вот такая задача, давай подумаем, как решить». Claude Code начинает спрашивать, предлагать варианты, восстанавливать ресурсы. Если сказать «я новичок, помоги мне» — поведёт за руку с нуля.

PRINCIPLE.03

Доступ есть — самовольство нет

У агента доступ ко всем папкам компьютера. Но он ничего не сделает без вашего разрешения. Каждое действие переспросит. Новичкам — стоит начать с «разбери папку Downloads», просто для калибровки мышления.

SEGMENT 04 // CLAUDE CODE vs CLAUDE.AI

Где живёт разница

Часто путают. Claude.ai в браузере и Claude Code в терминале — разные звери, хотя у обоих одна голова. Один — собеседник. Второй — исполнитель с руками.

// CLAUDE.AI (WEB)

Собеседник

  • Чат в браузере — claude.ai
  • Нет доступа к файлам компьютера
  • Не делает приложения, не устанавливает ничего
  • Не запускает подагентов
  • Нет MCP-интеграций со сторонними сервисами
  • Хорош для размышлений, текстов, разовых разборов
VERDICT: для разговора, не для производства
// CLAUDE CODE (CLI)

Исполнитель

  • Терминал на компьютере — claude-code
  • Полный доступ к файловой системе (с разрешением)
  • Делает приложения, проекты, сам деплоит
  • Разворачивает 30–45 подагентов параллельно
  • MCP-интеграции — Higgsfield, Figma, Premiere и т.д.
  • Промпты зашиваются в проект (CLAUDE.md, скиллы)
  • Для серьёзной автоматизации и реальных сервисов
VERDICT: для производства, не для болтовни
SEGMENT 05 // MCP & INTEGRATIONS

Подключаемся через MCP

MCP-сервер — это коннектор, который пристёгивает сторонний сервис прямо к агенту. После этого Higgsfield, Krea или Freepik живут внутри Claude. Картинки и видео генерятся прямо в чате, голосом.

Если раньше в Canva мы делали всё руками — теперь и в Canva руками делать необязательно. Всё может делать Claude по голосовому заданию.
